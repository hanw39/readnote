三种需求框架思路                                                                                                
  方案 A：以「减噪」为核心 ⭐ 推荐  

  围绕"信息过载"这个主要痛点，所有需求都服务于快速找到值得读的内容。                                                       
    
  - 文章搜索（按标题关键词）                                                                                               
  - 文章按时间过滤（--since 7d / --before 30d）                                                                            
  - 每个 blog 的未读数统计摘要（blogwatcher stats）
  - 批量按日期标为已读（消化积压）                                                                                         
  - Blog 健康状态检测（找出长期不更新的 blog，方便清理）
  - Edit 支持更多字段（--feed-url、--name、--scrape-selector）                                                             
                                         
  核心逻辑：先让现有内容易于管理，再考虑扩展。

  ---
  方案 B：以「Blog 生命周期」为核心

  把需求按 blog 的管理流程组织：发现 → 追踪 → 维护 → 清理。

  - OPML 导入/导出（批量迁移 blog 列表）
  - Blog 活跃度检测 + --stale 过滤
  - Edit 支持完整字段修改
  - 批量 remove（按分类批量删除）
  - 文章归档（超过 N 天自动标为已读）

  核心逻辑：把 BlogWatcher 做成一个可长期维护的订阅管理器。

  ---
  方案 C：以「CLI 完整性」为核心

  补全所有明显缺失的 CRUD 操作和查询能力，不分优先级。

  - Edit 完整字段、Rename blog、批量操作
  - 文章搜索、分页、排序
  - 统计命令、导出（JSON/CSV）
  - 健康检测、OPML

  核心逻辑：工具该有的都有，但可能优先级散。
